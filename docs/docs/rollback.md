# Rollback guide

Each OMA release is designed so rolling back is mechanically simple —
the v0.3a → v0.5 sequence introduces only additive ontology/DSL
features, no destructive schema changes.

## General principle

- Every schema change is **additive**. Rolling back a release never
  invalidates previously valid data; it only hides fields.
- Committed native files (`.mcp.json`, `kiro-agents/*.agent.json`) are
  deterministic functions of the DSL. Reverting the compiler and
  recompiling is enough.
- `.omao/triggers.json` is regenerated by `oma compile --all` and is
  safe to delete.

## v0.5 → v0.4

1. `git revert` (or selective checkout) the v0.5 commits.
2. Delete the four newly-added `plugins/*/` `.oma.yaml` files if their
   presence is causing downstream confusion. Native files emitted under
   `plugins/*/.mcp.json` can be removed since those plugins shipped
   without them before v0.5.
3. Drop the `--strict-enterprise` and `--enterprise` flags from CI and
   any operator scripts.
4. Run `oma compile --all` to regenerate triggers.

## v0.4 → v0.3b

1. `git revert` v0.4 commits affecting:
   - `schemas/ontology/deployment.schema.json` (SLSA artefact shape)
   - `schemas/harness/dsl.schema.json` (telemetry / policies bodies)
   - `tools/oma_audit/` (the JSONL appender)
   - `scripts/oma/{validate,migrate-audit}.sh`
   - `docs/docs/compliance/*.md`
2. If teams had already dual-write'd to `.omao/audit.jsonl`, keep the
   file — it stays readable by any later release.
3. `oma compile --all` is idempotent.

## v0.3b → v0.3a

1. Revert the `dsl.schema.json` v2 additions (version enum, workflows,
   telemetry/policies reserved slots, metadata).
2. Remove `_validate_workflows` and `_verify_policy_regexes` from
   `tools/oma_compile/compile.py`, along with the harness emission path
   (`_build_harness_rules`, `_build_hooks_json`, `_emit_harness`) that
   writes each plugin's `hooks/harness-rules.json`, `hooks/enforce.py`,
   and the managed `PreToolUse` entry in `hooks/hooks.json`.
3. `oma compile --all` still succeeds — v0.3a files are already v1 or
   they only added ontology fields.

## v0.3a → v0.2.0-preview.1

1. Revert the v0.3a commits. All six baseline schemas revert to their
   v0.2 shape; the four new `schemas/{audit,common,ontology/spec,ontology/adr}`
   files come out with the revert.
2. `.omao/triggers.json` falls back to the old `{keyword, route, plugin}`
   shape on the next `oma compile --all`. `hooks/user-prompt-submit.sh`
   read the new shape only — so if you are pinning the v0.2 hook, that
   is restored by the same revert.
3. Remove `pyproject.toml` if downstream tooling relied on the earlier
   "no Python dep file" setup; the runtime does not require it.

## CI rollback

Two workflows changed in v0.3a:

- `.github/workflows/oma-foundation.yml` — switched from ad-hoc
  `pip install jsonschema pyyaml pytest` to `pip install -e .[dev]`.
- `.github/workflows/validate-schemas.yml` — switched to `pip install -e .`.

Reverting those workflows alongside the commits is sufficient. The
underlying `foundation` and `validate-schemas` jobs keep succeeding
regardless of whether `pyproject.toml` is present, because the pinned
versions intersect with whatever pip resolves anyway.

## What cannot be rolled back cheaply

- **`tools.oma_audit.append` adoption.** Once skills switch to the
  helper, they validate every event against the schema. Rolling back
  the schema without reverting the skill changes leaves the skills
  writing valid-but-ignored records. Mitigation: treat `.omao/audit.jsonl`
  as append-only and keep the file across releases.
- **Committed SLSA provenance URIs.** The object form of
  `Deployment.artifact` is forward-only in deployed records.
  Post-rollback, the object form still validates under older schemas
  because v0.4 adds `oneOf: [string, object]` — the object form is
  simply opaque to the v0.3 compiler. No data loss.
